此README文件为一次代码复盘，分析自己在编写代码时遇到的问题。

作业的代码要求是用自己熟悉的程序设计语言实现对一个命题公式的：

（1）真值表的打印

（2）公式类型的判断

（3）两种主范式的求解

抛开代码运行环境这个繁琐、暧昧的问题，我在编程过程中遇到的最困难的事就是把问题拆解成几个部分来实现。这件事想起来简单，做起来却因拆解问题的思路复杂而变得困难。

例如，在思考如何实现第一个问题时，我只考虑了打(真值)表，仅仅将每种变元的取值呈现在输出结果上面，而变元取值无法在后续求解问题时使用，且打表还花费了我很多时间，实属不明智了。

此外，我遇到的问题还有很多：

1.把每一个命题公式的计算想得太简单了，没有考虑到设计公式类型的优化转换(中缀表达式(如p&q)转后缀表达式(如pq&))、公式去空格和计算公式真值的算法，导致花费了较多不必要的时间。

2.对一些临界条件进行了重复判断，比如，判断变元是否是小写字母，本来仅仅 小写&&字母 即可结束判断，我却添加了 !小写--> return false 的if判断，是否提升代码健壮性不说，至少写法是不够简洁的。

3.求解第三个问题时，由于自己对主范式的理解不够深刻，最初写代码时没有考虑求出极小/大项，本来思路就混乱，该部分的代码更是基本上写不出来。

4.在几个小时后，我写好了所需的类和函数，准备在main函数中调用。可是，在设计函数时我并没有仔细考虑函数的返回值，基本上都是默认写的int导致我在面对运行报错时不知道问题在哪。

5.除4.以外，我遇到了一个我至今无法理解的问题：一个函数被定义在类中，它的结尾包含了两个之后会在类中被定义的函数的声明。在main函数中仅调用该函数时，代码可以实现这三个函数的功能，但是，如果不将那两个函数的声明放在该函数的结尾，在调用该函数时只能实现这一个函数的功能；此外，如果我在未添加声明的情况下在main函数中调用了这三个函数，运行代码时会报错。

其次，是分析要使用的数据结构。

考虑打表时，我最先想到的是使用n*（2^n）的二维数组，但是会出现表头元素重复的情况。后面看了ds的代码，决定采用 set去重 + Vector来构建真值表的框架，一来去重了，二来Vector见得少，可以在写的过程中熟悉一下。

设置运算符优先级时，我首先考虑到了栈，虽然我对栈的语法极不熟悉，看了很久ds的代码还是不太懂，但我最终还是迫于知识储备不足选择了栈，如今我也意识到：栈不只是用于优先级的设置，还可以用于检查括号匹配、存值（一些情况下）和表达式的形式转换（如中缀转前缀）。

判断公式类型时，由于在第一个问的求解过程中用到了Vector，在计算每一行赋值条件下命题公式的真值时可以使用二维Vector结合后缀表达式便利公式真值的计算。

求解公式主范式时也使用了Vector，就不提及细节了。

提及本次作业的代码运行环境，我使用的是自以为最熟悉（仅仅是安装了许多插件，为IDE配置了一个图片背景，亲手过写代码的次数屈指可数）的VsCode IDE。

但在编译运行代码时还是出现了很多我意想不到的意外，

1.在VsCode上运行一个文件的前提是编译文件得到一个exe文件，这点是我未曾注意到的。如果出现编译文件存在但文件无法运行的情况，往往可以通过删除编译文件重新进行编译得到编译文件。

2.Windows的cmd终端默认编码格式为gbk，而VsCode终端的默认编码格式为utf-8，如果在VsCode上的运行输出内容有乱码，将VsCodeIDE的编码格式改为gbk，更改后中文注释很可能乱码，此时键盘输入ctrl+z注释内容可恢复正常，然后重新运行代码即可，此时不会再出现乱码的情况。

3.我在VsCode上安装了代码补全的辅助工具，在独立编程的过程中会干扰我的编写思路，甚至有时补全的代码并不是正确的代码，这个情况稍有些恼人。

4.此本地代码仓库已链接至远程仓库，但我并不是很熟悉git的流程，在VsCode上提交代码时也总是忘记一些细节（比如，每次需要填写提交的备注内容），还需练习。
